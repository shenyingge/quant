# Coding Rules

## 文件布局

- `src/` 根目录不保留 Python 文件，必须进入子目录。
- 业务实现必须进入子目录，不再新增根级业务 Python 文件。
- 测试文件放在 `tests/unit/`、`tests/integration/`、`tests/live/` 等分类目录。

## 命名

- 类名使用 `PascalCase`。
- 函数、方法、文件名使用 `snake_case`。
- 常量使用 `UPPER_SNAKE_CASE`。
- 私有方法使用前导下划线。

## 导入

- 标准库、第三方、本地模块分组。
- 优先使用绝对导入。
- 避免循环导入；出现边界问题时先调整模块职责。

## 类型与接口

- 公共函数必须有输入和返回类型标注。
- 对外结构优先保持显式字段，不隐藏关键来源和状态。
- 改接口时要同步调用方、测试和相关文档。

## 配置

- 配置统一走 `src/infrastructure/config/`（`settings`）和环境变量。
- 不允许在业务代码里硬编码环境相关值。
- 新增配置时同步更新 `.env.example`。

## 错误处理与日志

- 记录带上下文的错误日志。
- 明确区分可恢复告警与真正失败。
- 不在日志中输出敏感信息。

## 测试

- 优先补最靠近改动的单元测试。
- 需要跨组件验证时再补集成测试。
- 修 bug 时至少覆盖一个回归场景。

## 数据与迁移

- 数据库结构变化必须配套 migration。
- migration 命名遵循 `YYYYMMDD_HHMMSS_hash_description.py`。
- 不直接改线上事实数据，除非用户明确要求。
